# 拉野计算规则 v1

本文档定义当前“拉野”事件的自动候选计算口径。拉野不是纯硬事实事件，数据库能较稳定证明“某一路线兵与真正中立野怪发生交战”，但不能总是证明具体是哪名英雄完成了拉野动作。因此当前版本将“拉野成立”和“可能执行者”分开处理。

## 1. 事件定义

在前期对线阶段，某一方边路线上小兵与真正中立野怪发生连续伤害或死亡交互，说明该方该路兵线被野怪牵引，记为该方该路发生“拉野”。

示例：

- 天辉下路小兵和中立野怪打起来：记为“天辉下路拉野”。
- 夜魇上路小兵和中立野怪打起来：记为“夜魇上路拉野”。

这里的“天辉/夜魇哪一路”由线兵阵营和线兵所属路线决定，不直接由英雄决定。

## 2. 数据来源

主要数据源：

- `combat_logs`
  - `time`
  - `log_index`
  - `type`
  - `attackername`
  - `targetname`
  - `sourcename`
  - `targetsourcename`
  - `value`
  - `x`
  - `y`

- `dota_model_neutral_siege_creep`
  - `type`
  - `unit`
  - `team`
  - `ehandle`
  - `x`
  - `y`
  - `hp`
  - `lifeState`
  - `ownerClass`

辅助数据源：

- `player_intervals2`
  - 用于查找拉野地点附近英雄。
  - 只作为可能执行者/附近英雄证据，不作为拉野成立的核心证据。

## 3. 基础筛选

时间范围：

- 默认只看前 15 分钟：`0 <= time <= 900`。

路线范围：

- 只看边路：
  - 上路 `top`
  - 下路 `bot`
- 中路 `mid` 不输出拉野。

线兵定义：

- 天辉线兵：
  - `npc_dota_creep_goodguys_melee`
  - `npc_dota_creep_goodguys_ranged`
  - `npc_dota_creep_goodguys_flagbearer`
  - `npc_dota_goodguys_siege`

- 夜魇线兵：
  - `npc_dota_creep_badguys_melee`
  - `npc_dota_creep_badguys_ranged`
  - `npc_dota_creep_badguys_flagbearer`
  - `npc_dota_badguys_siege`

真正中立野怪定义：

- 单位名是 `npc_dota_neutral_*`。
- 并且在 `dota_model_neutral_siege_creep` 中满足：
  - `type = neutral_status`
  - `team = 4`
  - `ownerClass` 为空，或不是英雄控制来源。

这个过滤用于排除 Lycan、Chen、Enchantress、支配头盔等控制的“中立单位”。

## 4. 单条证据规则

一条 `combat_logs` 记录进入拉野候选证据，需要满足：

- `type` 是：
  - `DOTA_COMBATLOG_DAMAGE`
  - 或 `DOTA_COMBATLOG_DEATH`
- 交互双方一边是线上小兵，一边是真正中立野怪。
- 方向不限：
  - 线兵打野怪
  - 野怪打线兵
  - 线兵击杀野怪
  - 野怪击杀线兵
- 排除英雄、召唤物、幻象、英雄控制野怪和普通野区打野。

## 5. 路线判断

路线判断优先级：

1. 优先用 `dota_model_neutral_siege_creep` 中线兵 `ehandle` 的轨迹判断路线。
   - 同一只线兵在前期多数时间所在的路线，即为该线兵所属路线。
2. 如果无法匹配 `ehandle`，使用交互点或死亡点坐标兜底：
   - `y - x > 35`：上路
   - `x - y > 35`：下路
   - 其他：中路
3. 判断为中路时，不输出拉野。

## 6. 阵营判断

阵营由线兵决定：

- `goodguys` 线兵：天辉该路拉野。
- `badguys` 线兵：夜魇该路拉野。

注意：这里表示“该方该路线兵被拉野”，不等于一定能证明该方某个英雄执行了拉野。

## 7. 聚合规则

候选证据按以下维度聚合：

- 同一线兵阵营
- 同一路线
- 相近时间
- 相近位置/野区区域

当前参数：

- 时间间隔：`<= 35s` 归为同一组。
- 位置距离：`<= 28 raw` 归为同一组。
- 如果 `combat_logs` 缺少坐标，则用线兵/野怪状态表在相近时间、相近位置的匹配结果补坐标。

输出门槛：

- 同组交互记录数 `>= 3`，或
- 同组线兵/野怪死亡记录数 `>= 2`。

满足任一门槛，输出一条“拉野”事件。

## 8. 英雄归因

拉野成立不强依赖英雄归因。

当前只用 `player_intervals2` 找附近英雄：

- 时间窗口：拉野组开始前后，默认前后各 `8s`。
- 空间范围：拉野中心 `1500` 码内。
- 天辉、夜魇附近英雄都会输出。

输出解释：

- `heroes` 字段记录附近英雄。
- `结果` 字段明确写“某方某路小兵与中立野怪发生连续交战，判定为某方某路拉野”。
- 不直接写“某英雄拉野成功”，除非后续加入更强的英雄行为证据。

## 9. 输出格式

标签：

- `拉野`

置信度：

- 当前写为“未知”。
- 后续通过人工批准样本做假设检验，验证该口径的 precision/recall 后再确定模型置信度。

结果示例：

```text
天辉下路小兵与中立野怪发生连续交战，判定为天辉下路拉野；交互数 2，线兵死亡 1，野怪死亡 1；附近天辉英雄：先知、树精卫士；附近夜魇英雄：森海飞霞
```

证据示例：

```text
pull_interaction log_index=90385 type=DOTA_COMBATLOG_DEATH lane_unit=npc_dota_creep_goodguys_ranged neutral_unit=npc_dota_neutral_kobold direction=lane_to_neutral lane=bot xy=(161.6,85.6) value=38 lane_status_ehandle=2345468 neutral_status_ehandle=11602251
```

## 10. 需要规避的问题

1. 控制野怪误判
   - Lycan、Chen、Enchantress、支配头盔等会让 `npc_dota_neutral_*` 变成英雄控制单位。
   - 必须用 `team = 4` 和 `ownerClass` 过滤。
   - 如果 `sourcename` 或 `targetsourcename` 指向英雄，也需要排除。

2. 普通打野误判
   - 英雄打野怪不算拉野。
   - 必须出现线兵和真正中立野怪交互。

3. 召唤物误判
   - 树人、狼、幻象、召唤物和野怪/线兵交战，不作为拉野核心证据。

4. 中路误判
   - 中路线兵和中立单位异常交互，不输出拉野。

5. 坐标缺失
   - 很多 `DOTA_COMBATLOG_DAMAGE` 没有 `x/y`。
   - 需要用死亡日志和 `dota_model_neutral_siege_creep` 状态表补坐标。

6. 同名野怪多处存在
   - 只用 `npc_dota_neutral_*` 名字会有歧义。
   - 需要结合时间、位置和状态表匹配。

7. 单条死亡噪声
   - 单个线兵或野怪死亡不一定代表完整拉野。
   - 需要使用交互数或死亡数门槛。

8. 执行英雄不确定
   - 数据能证明兵线和野怪打起来，但不一定能证明是谁拉的。
   - 附近英雄只能作为候选，不作为硬结论。

9. 断线/勾兵混淆
   - 断线/勾兵看的是线兵与敌方线兵、英雄、经验的关系。
   - 拉野看的是线兵与真正中立野怪的交互。
   - 只要出现真正中立野怪交互，优先进入拉野口径。

## 11. 一句话规则

`0-15分钟内，边路线兵与 team=4 的真正中立野怪发生连续伤害或死亡交互；按线兵阵营和线兵所属路线判断为天辉/夜魇某一路拉野；同组交互数>=3或死亡数>=2才输出，附近英雄只作为可能执行者记录。`
