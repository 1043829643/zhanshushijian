# 战术事件识别项目现状与目标

## 项目目标

本项目的目标是把 Dota2 比赛数据库中的原始/半原始数据，稳定复算成可人工复核、可追溯、可迭代的战术事件表。

最终希望达到：

1. 每一条战术事件都能说明“为什么被识别出来”，并保留对应证据。
2. 每一种事件口径都有清晰的数据来源、触发条件、阈值、合并规则和人工复核点。
3. 输出 Excel 与人工标注案例保持同款结构，方便运营或分析人员修改。
4. 不在单条事件里主观填写置信度；置信度改为口径模型质量，用人工批准样本做假设检验。
5. 通过多局人工批准样本持续校准规则，使结果准确、可控、可解释。

## 当前状态

当前仓库已经包含三类内容：

1. 事件定义与算法规则文档。
2. 从 StarRocks 抽取单局数据、计算事件、导出 Excel 的脚本。

## 已落地的重要口径

### 对线分路

读取 `player_intervals2` 中每个英雄 `0 <= time <= 300` 的位置。

路线判断：

```text
y - x > 35 => top
x - y > 35 => bot
其他 => mid
```

统计每个英雄 5 分钟内落在 top/mid/bot 的快照数，快照最多的路线作为主分路。

阵营映射：

```text
天辉：bot=优势路，top=劣势路，mid=中路
夜魇：top=优势路，bot=劣势路，mid=中路
```

每个英雄输出一条 `对线分路` 事件，时间范围固定 `00:00-05:00`。

### 对线区

`对线区` 是 `断线` 和 `勾兵` 共用的区域口径。

定义：

```text
某一路在一塔存活阶段，双方线上小兵正常交汇、交战、死亡的主要区域。
```

实现：

1. 从 `dota_model_neutral_siege_creep` 按 `ehandle` 跟踪线上小兵。
2. 用前 10 分钟小兵死亡点生成每一路的 `lane_zone`。
3. 只把双方一塔存活、死亡点附近存在敌方同路线小兵的死亡点作为正常交汇样本。
4. 每一路取样本坐标的 10%-90% 分位，再外扩 10 raw 坐标。

若某局没有 `dota_model_neutral_siege_creep`，则不能可靠确认 `断线/勾兵`。

### 断线

前 10 分钟内，若某阵营某一路小兵满足：

```text
该阵营该路线一塔存活
小兵死在 lane_zone 外
500 范围内无敌方线上小兵
击杀来源可映射为敌方英雄
```

则作为断线证据。

同一英雄、同一路线、10 秒内连续触发 `>=2` 或 `>=3` 个符合规则的小兵死亡，输出一条 `断线` 事件。

当前实现默认使用 `>=2`，后续需要通过人工批准样本比较 `>=2` 和 `>=3` 的 Precision/Recall。

### 勾兵

前 10 分钟内，若某阵营某一路小兵满足：

```text
该阵营该路线一塔存活
小兵死在 lane_zone 外
500 范围内有敌方线上小兵
敌方英雄获得该小兵死亡经验
```

则输出 `勾兵` 事件。

`结果` 字段以经验获得为准，记录获得经验的敌方英雄。

### 置信度

最终事件 Excel 不输出 `confidence` 列。

置信度改为口径模型质量指标，通过人工批准样本做假设检验：

```text
model_confidence = 0.35 * Precision
                 + 0.35 * Recall
                 + 0.20 * 字段准确率
                 + 0.10 * 人工批注命中率
```

每一种事件口径分别检验 Precision、Recall、字段准确率和人工批注命中率。

## 当前局限

1. 战斗聚类仍偏召回，`小规模冲突` 可能偏多，需要继续用人工批准表压误报。
2. `守塔团`、`高地团`、`肉山团`、`魔晶团` 等特殊上下文标签还没有完全接入最终计算。
3. `拉野`、`蹲人` 仍主要停留在规则定义阶段，自动化输出需要继续实现。
4. `断线/勾兵` 强依赖 `dota_model_neutral_siege_creep`，历史比赛如果没有该表，只能降级。
5. 目前还缺少自动化评估脚本，把模型输出和人工批准 Excel 批量对齐并计算假设检验结果。

## 下一阶段目标

优先级从高到低：

1. 建立人工批准表对比脚本。
   - 输入：模型 Excel、人工批准 Excel。
   - 输出：TP/FP/FN、字段错误归因、人工批注命中情况。

2. 对每个口径做假设检验。
   - 先验证 `对线分路`、`控符`、`肉山击杀`、`断线/勾兵`。
   - 再验证 `小规模冲突/团战`。

3. 收敛战斗聚类。
   - 减少对线期消耗误报。
   - 合并相邻重复战斗。
   - 优化参战英雄过滤。

4. 落地特殊战斗标签。
   - `守塔团`
   - `高地团`
   - `肉山团`
   - `肉山尝试`
   - `魔晶团`

5. 补齐对线操作标签。
   - `拉野`
   - `蹲人`
   - 对 `断线/勾兵` 做多局阈值校准。

6. 固化交付流程。
   - 一条命令完成抽取、计算、导出、评估。
   - 所有输出都带版本号、规则版本和证据路径。

## 常用流程

设置数据库环境变量：

```powershell
$env:DB_HOST='47.86.96.51'
$env:DB_PORT='9030'
$env:DB_USER='dota2_reader'
$env:DB_PASS='******'
$env:DB_NAME='dota2_analysis'
```

抽取一局：

```powershell
python work/pipeline_extract_match.py 8826042346 --dt 2026-05-26
```

计算事实事件：

```powershell
python work/pipeline_compute_fact_events.py raw_db/match_8826042346
```

计算战斗事件：

```powershell
python work/pipeline_compute_fight_records.py raw_db/match_8826042346
```

导出 Excel：

```powershell
node work/pipeline_build_tactical_table.mjs 8826042346
```

## 关键文件

- `work/pipeline_extract_match.py`：从 StarRocks 抽取比赛数据。
- `work/pipeline_compute_fact_events.py`：计算事实类、资源类、对线异常类事件。
- `work/pipeline_compute_fight_records.py`：计算战斗聚类与战斗事件。
- `work/pipeline_build_tactical_table.mjs`：合并事件并导出 Excel。
- `work/lane_zone_definition.md`：对线区定义。
- `work/hypothesis_testing_confidence.md`：口径置信度假设检验方案。
- `outputs/dota2_tactical_event_recognition_algorithms_v1.9_20260609.xlsx`：当前算法规则表。
