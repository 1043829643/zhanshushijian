# 当前战术事件计算方法清单

本文档整理当前工作区代码中已经实现或部分实现的战术事件计算方法。清单按当前代码口径描述，用于后续核对、补足和调整计算规则。

## 一、事实类事件

### 1. 对线分路

- 数据源：`player_intervals2`
- 时间范围：`0 <= time <= 300`
- 计算逻辑：
  - 读取每个英雄 0-5 分钟内的 `x/y` 位置。
  - 用原始坐标判断路线：
    - `y - x > 35`：上路 `top`
    - `x - y > 35`：下路 `bot`
    - 其他：中路 `mid`
  - 对每个英雄统计 5 分钟内落在 `top/mid/bot` 的快照次数。
  - 快照最多的路线为主分路 `primary_lane`。
  - 将快照次数折算成秒数：`lane_seconds = count * (300 / 总有效样本数)`。
  - 非主路线停留时间 `>= 60s` 时，记录为游走/次要路。
- 阵营映射：
  - 天辉：`bot = 优势路`，`top = 劣势路`，`mid = 中路`
  - 夜魇：`top = 优势路`，`bot = 劣势路`，`mid = 中路`
- 输出：
  - 每个英雄输出一条“对线分路”事件。
  - 时间范围固定为 `00:00-05:00`。
- 置信度：当前写为“未知”。

### 2. 出门抢远程兵/近战兵

- 数据源：
  - `combat_logs`
  - 小兵状态数据
- 时间范围：`0 <= time <= 31`
- 基础筛选：
  - 只看线上小兵死亡事件：`DOTA_COMBATLOG_DEATH`。
  - 被击杀目标必须是线上远程兵或近战兵。
  - 只识别上路/下路；如果来源路或坐标兜底结果为中路，直接跳过。
  - 远程兵输出为 `出门抢远程兵`，近战兵输出为 `出门抢近战兵`。
- 路线判断：
  - 优先用小兵状态表按 `ehandle` 的出生/早期轨迹判断来源路 `source_lane`。
  - 缺少 `ehandle` 时，用死亡点坐标兜底：
    - `y - x > 35`：上路
    - `x - y > 35`：下路
    - 其他：中路
- 金钱判断：
  - 在该死亡日志前后 `1s` 的相邻日志块里读取 `DOTA_COMBATLOG_GOLD`。
  - 只保留正数金钱。
  - 排除开局发钱 `gold_reason = 17` 和负数扣钱。
  - 金钱获得者通常就是补到这个兵的人；如果没有金钱日志，则用死亡日志中的击杀来源作为补刀来源兜底。
- 经验判断：
  - 在该死亡日志前后 `1s` 的相邻日志块里读取 `DOTA_COMBATLOG_XP`。
  - 只保留正数经验。
  - 输出所有分到该兵经验的英雄及经验值。
- 输出：
  - `heroes` 记录该兵相关的金钱获得者、经验获得者、击杀来源英雄。
  - `结果` 写清楚是哪一方哪一路的远程/近战兵死亡、谁补到/获得金钱、谁分到经验。
  - 每个死亡小兵输出一条事件，不做聚合。
- 置信度：当前写为“未知”。

### 2. 囤野/堆野

- 数据源：`player_intervals2`
- 计算逻辑：
  - 按英雄和时间排序读取快照。
  - 比较相邻快照中的 `camps_stacked` 和 `creeps_stacked`。
  - 如果任一计数器增加，则输出一次“囤野/堆野”事件。
- 输出：
  - 记录触发英雄、发生时间、增加的营地数和小兵数。
- 置信度：`0.86`

### 3. 断线

- 数据源：
  - `combat_logs`
  - 小兵状态数据
  - 防御塔状态数据
- 时间范围：前 10 分钟，`0 <= time <= 600`
- 基础筛选：
  - 只看小兵死亡事件。
  - 被击杀目标必须是线兵。
  - 死亡小兵需要能识别阵营和路线。
- 关键条件：
  - 这一路双方一塔仍存活。
  - 路线按小兵来源路 `source_lane` 判定；优先用 `ehandle` 出生/早期轨迹验证，缺少 `ehandle` 时才用死亡坐标兜底。
  - 小兵死亡点在本路“对线区”之外。
  - 小兵死亡点不在本路双方任一存活一塔保护区内；一塔保护区半径暂定 `12 raw`，塔下补进塔兵、塔杀小兵不算断线。
  - 死亡点附近没有敌方线兵。
  - 击杀来源是敌方英雄。
- 聚合规则：
  - 按同一英雄、同一路线、同一被断兵方聚合。
  - `10s` 内连续触发 `>= 2` 个符合条件的小兵死亡，输出一条“断线”事件。
- 输出：
  - 记录断兵英雄、断兵英雄 0-5 分钟主对线路、被断兵方、小兵来源路线、小兵数量、发生时间段和证据。
  - 如果中路英雄到下路断下路兵，结果写成“某英雄（中路）在下路断某方下路小兵”。
- 置信度：当前写为“未知”。

### 4. 勾兵

- 数据源：
  - `combat_logs`
  - `dota_model_neutral_siege_creep`
  - `player_intervals2`
  - 经验日志
  - 防御塔状态数据
- 时间范围：仅前 10 分钟，`0 <= time < 600`；`time >= 600` 不输出勾兵。
- 基础筛选：
  - 优先扫描有 `ehandle` 的线上小兵逐秒轨迹；缺少 `ehandle` 的小兵不能用轨迹法自动确认。
  - 小兵需要能识别阵营和来源路线。
  - 轨迹法的来源路线使用该 `ehandle` 全段主路线 `lane_by_ehandle`，避免刚出生靠近基地时被早期坐标误判成中路。
- 关键条件：
  - 为每个阵营、每条来源路线建立默认兵线轨迹：取同阵营同路线小兵在前 10 分钟内、仍处于该路线粗略区域内的高密度位置格子。
  - 小兵连续至少 `4` 个状态点偏离默认轨迹 `>= 650` 码。
  - 其中至少 `2` 个状态点的移动方向朝向 `900` 码内的敌方英雄、敌方线兵或中立野怪。
  - 轨迹片段必须发生在该小兵首次正常接触同路线敌方线兵之前，允许 `2s` 容差。
  - 若同一时间段同阵营同路线至少 `2` 个小兵同步偏离，证据更强。
  - 最近触发英雄按轨迹片段附近 `1500` 码内最近的敌方英雄归因；找不到英雄时保留为弱轨迹候选。
  - 原死亡/经验口径继续作为补证据：小兵在对线区外死亡、附近有敌方线兵且敌方英雄获得经验时，也进入勾兵候选。
- 聚合规则：
  - 轨迹候选按最近触发英雄组合、被勾兵方和来源路线聚合。
  - 死亡/经验候选按获得经验英雄组合、被勾兵方和来源路线聚合。
  - 同 key 候选在 `10s` 内合并成一条事件；事件结束时间取轨迹片段或死亡组的最后时间。
- 输出：
  - 记录勾兵英雄或获得经验英雄、其 0-5 分钟主对线路、被勾兵方、小兵来源路线、小兵数量和时间段。
  - `evidence` 写入 `trajectory_score`、`max_lane_offset`、`off_path_seconds`、`toward_enemy_seconds`、`linked_creep_count`、`nearest_enemy_hero`。
- 置信度：当前写为“未知”。

### 5. 拉野

- 数据源：
  - `combat_logs`
  - `dota_model_neutral_siege_creep`
  - `player_intervals2`
- 时间范围：前 15 分钟，`0 <= time <= 900`
- 基础筛选：
  - 只看边路，不看中路。
  - 只看线上小兵和真正中立野怪的交互。
  - 交互证据包括两类：
    - `combat_logs` 中线上小兵与中立野怪的 `DOTA_COMBATLOG_DAMAGE` 或 `DOTA_COMBATLOG_DEATH`。
    - 以 combat 直接交互为锚点，在锚点前 `45s` 到后 `5s` 内，`dota_model_neutral_siege_creep` 里同一路线线兵与同名中立野怪距离 `<= 550` 码的状态贴近证据。
    - 如果没有 combat 直接交互锚点，也全局扫描状态表：同一路线边路线兵与真正中立野怪距离 `<= 550` 码，并形成连续掉血片段，也进入拉野候选。
    - 实现时用 raw 坐标计算距离后换算成游戏码，`550` 码约等于 `4.23 raw`。
- 真正中立野怪过滤：
  - 单位名是 `npc_dota_neutral_*`。
  - 在 `dota_model_neutral_siege_creep` 中满足 `type = neutral_status` 且 `team = 4`。
  - 排除 `ownerClass` 指向英雄控制来源的中立单位。
  - 如果 `sourcename` 或 `targetsourcename` 指向英雄，也排除，避免 Lycan、Chen、Enchantress、支配头盔等控制野怪误判。
- 路线判断：
  - 拉野优先用线兵 `ehandle` 全段轨迹的主路线判断所属路线；如果主路线缺失，再用出生/早期轨迹和当前坐标兜底。
  - 无法匹配时，用交互点/死亡点坐标兜底：
    - `y - x > 35`：上路
    - `x - y > 35`：下路
    - 其他：中路，中路不输出拉野。
  - 如果 `dota_model_neutral_siege_creep` 缺少 `ehandle`，仍使用同秒状态点参与坐标匹配和贴近判断，但不把单点状态误写成出生路线轨迹。
- 阵营判断：
  - `goodguys` 线兵：天辉该路拉野。
  - `badguys` 线兵：夜魇该路拉野。
- 聚合规则：
  - 按同一线兵阵营、同一路线、相近时间、相近位置聚合。
  - 时间间隔 `<= 35s` 归为同一组。
  - 位置距离 `<= 28 raw` 归为同一组。
  - 同组交互记录数 `>= 3`，或同组线兵/野怪死亡记录数 `>= 2`，才输出一条“拉野”事件。
  - 只有状态贴近而完全没有 combat 直接交互锚点时，必须额外满足连续秒数和线兵/野怪掉血门槛；否则不输出拉野，避免把线兵路过野点误判为拉野。
  - 状态贴近的掉血判断优先按真实 `ehandle` 跟踪同一单位；如果状态表缺少真实 `ehandle`、只能得到 `synthetic` 临时句柄，则按“单位类型 + 近似位置轨迹”兜底估算连续掉血，避免 08:20 这类线兵与野怪持续交战但 ehandle 每秒变化的片段被误过滤。
- 英雄归因：
  - 先确认“这一波野怪被成功拉野”：英雄触发野怪仇恨，野怪被英雄带离营地，随后野怪勾到线兵仇恨，线兵被带走，最后线兵与野怪发生连续交战或 `550` 码内贴近停滞。
  - 拉野成立后，再反推执行者；执行者不是后续补刀/收野的人，也不是单纯站在附近的人。
  - 在该波拉野移动最早阶段，使用 `combat_logs` 找这组野怪与英雄的最早仇恨交互：谁打了这组野怪，或这组野怪最早打到谁。
  - 当前实现采用“英雄主动打到该组野怪”或“该组野怪最早打到英雄”的第一条日志作为强证据；后续会继续补全“英雄走近拉仇恨但未发生伤害”的移动归因。
  - 野怪仇恨触发窗口：拉野状态组开始前 `12s` 到开始后 `10s`。
  - 没有英雄-野怪仇恨交互证据时，不再用位置兜底输出拉野执行者，避免把附近英雄误算为拉野人。
  - 同一波拉野若在 `8s` 内拆出多条候选，按最早的野怪仇恨触发时间和日志序号只保留一条。
  - 另用 `player_intervals2` 找拉野中心 `1500` 码内、事件前后 `8s` 内的附近英雄，作为环境证据写入结果列。
- 输出：
  - `heroes` 记录推断出的拉野执行者；结果列记录某方某路小兵与中立野怪连续交战、交互数、线兵死亡数、野怪死亡数和附近英雄。
- 置信度：当前写为“未知”。
- 详细规则文档：`拉野计算规则_v1.md`

### 6. 控符

- 数据源：`match_chat_events`
- 事件类型：
  - 拾取神符
  - 装瓶神符
  - 反补神符
- 符类型：
  - 神符
  - 赏金符
  - 水符
  - 智慧符
  - 其他可识别资源符
- 计算逻辑：
  - 读取聊天事件中的符类事件。
  - 识别处理符的英雄、符类型和动作。
  - 对河道神符，读取处理符英雄事件时刻的位置，靠近上符点写“上河道”，靠近下符点写“下河道”。
  - `value=5` 的赏金神符单独走赏金符点逻辑，不套普通河道符逻辑；0 分钟赏金符按 4 个点判断：
    - 上河道
    - 下河道
    - 天辉赏金符点：server 坐标 `(590, -4637)`，按当前 homography 换算为 raw 坐标约 `(133.6, 90.4)`。
    - 夜魇赏金符点：server 坐标 `(-1000, 4443)`，按当前 homography 换算为 raw 坐标约 `(120.0, 166.8)`。
  - 赏金神符根据处理符英雄事件时刻最近的 4 个赏金符点，写成“上河道/下河道/天辉赏金符点/夜魇赏金符点”。
  - `value=8` 的智慧符单独走智慧符点逻辑，不套普通河道符逻辑：
    - 天辉智慧符点：raw 坐标约 `(66.0, 134.0)`，server 坐标约 `(-7468, 610)`。
    - 夜魇智慧符点：raw 坐标约 `(191.0, 118.0)`，server 坐标约 `(7537, -1285)`。
  - 智慧符根据处理符英雄事件时刻最近的 2 个智慧符点，写成“天辉智慧符点/夜魇智慧符点”。
  - 结合符点附近英雄位置，补充双方在符点附近的存在情况。
  - 对“装瓶后又拾取”的重复记录做去重。
- 输出：
  - 记录英雄、动作、符点、符类型、时间和附近双方英雄，例如“风暴之灵罐装上河道恢复神符”“森海飞霞拾取天辉赏金符点赏金神符”。
- 置信度：`0.90`

### 7. 开雾

- 数据源：`combat_logs`
- 识别依据：
  - 使用 `item_smoke_of_deceit`
  - 英雄获得 `modifier_smoke_of_deceit`
- 计算逻辑：
  - 以一次开雾使用为起点。
  - 在后续窗口内收集同队进入雾状态的英雄。
  - 如果同队出现下一次开雾，则用下一次开雾截断当前窗口。
- 输出：
  - 记录开雾使用者、进入雾状态的英雄列表和证据日志。
- 置信度：`0.78`

### 8. 肉山击杀

- 数据源：
  - `combat_logs`
  - `match_chat_events`
- 计算逻辑：
  - 从战斗日志识别 Roshan 死亡。
  - 结合后续掉落或获得事件，如不朽盾、奶酪、刷新碎片、战旗等。
  - 判断击杀方和资源归属。
- 输出：
  - 记录击杀方、击杀时间、掉落物和获得英雄。
- 置信度：`0.96`

### 9. 肉山尝试

- 数据源：`combat_logs`
- 时间范围：`time >= 600`
- 计算逻辑：
  - 只看对 Roshan 造成伤害的记录。
  - 连续 Roshan 伤害按时间间隔聚合，当前间隔阈值约为 `45s`。
  - 聚合窗口内总伤害需要 `>= 1500`。
  - 如果该窗口后很快完成 Roshan 击杀，则不输出“肉山尝试”，交给“肉山击杀”事件。
  - 如果双方都对 Roshan 造成伤害，则结果描述为双方尝试/争夺。
- 输出：
  - 记录尝试方、时间段、总伤害和参与英雄。
- 置信度：`0.72`

### 10. 资源获得

- 数据源：`match_chat_events`
- 计算逻辑：
  - 识别通用资源获得类聊天事件。
  - Roshan 掉落类资源不单独输出，因为已经并入“肉山击杀”。
- 输出：
  - 记录资源类型、获得方、获得英雄和时间。
- 置信度：`0.90`

### 11. 带盾阵亡

- 数据源：
  - `match_chat_events`
  - `combat_logs`
- 计算逻辑：
  - 先记录英雄获得不朽盾的时间。
  - 如果同一英雄在不朽盾有效期内死亡，则输出“带盾阵亡”。
  - 当前不朽盾有效期按约 `300s` 处理。
- 输出：
  - 记录带盾英雄、死亡时间和相关证据。
- 置信度：`0.88`

## 二、战斗类事件

战斗类事件先从 `combat_logs` 中抽取英雄间伤害、控制和死亡信号，再结合 `player_intervals2` 补充位置，用时间和空间规则聚合为一次战斗。

### 通用战斗信号

- 伤害信号：
  - 类型：`DOTA_COMBATLOG_DAMAGE`
  - 条件：英雄对英雄造成伤害，攻击者和目标不同。
  - 分段伤害阈值：
    - `0 <= time < 600`：伤害 `>= 30` 才作为战斗伤害信号。
    - `time >= 600`：伤害 `>= 45` 才作为战斗伤害信号。
  - 这样前 10 分钟可以保留较小规模的对线/边路换血信号；10 分钟后维持旧阈值，避免中后期远端消耗过多进入战斗候选。
- 死亡信号：
  - 类型：`DOTA_COMBATLOG_DEATH`
  - 条件：死亡目标是英雄
- 控制信号：
  - 类型：`DOTA_COMBATLOG_MODIFIER_ADD`
  - 条件：英雄对英雄施加控制，控制时长 `>= 0.25s`
  - 控制时长来自眩晕、减速、沉默、缠绕、位移控制等字段或标记
- 忽略幻象：
  - 攻击者或目标为幻象时，不作为核心战斗信号。

### 通用战斗聚合

- 聚合条件：
  - 分段时间间隔：
    - `0 <= time < 600`：相邻战斗信号时间间隔 `<= 7s` 才继续并入同一候选。
    - `time >= 600`：相邻战斗信号时间间隔仍为 `<= 12s`，计算方式不变。
  - 且位置信号距离不超过约 `48` 原始坐标单位
- 事件保留条件：
  - 形成 N 抓 1 人数结构后，看“被抓的那一个人”承受的非远程主战斗伤害，而不是看双方总伤害。
  - 被抓目标承受伤害 `>= 600`，或
  - 被抓目标死亡。
  - 被抓目标未死亡且承伤 `< 600` 的 2v1 消耗，不输出为 GANK。
- 当前已经移除旧兜底：
  - “信号数 >= 2 且包含控制”不再作为保留条件。
- 重叠/近邻战斗合并：
  - 两个候选战斗如果时间重叠，或间隔 `<= 3s`，不再只看整段战斗中心距离。
  - 对每个候选战斗，在每一秒读取该候选双方英雄本体在 `player_intervals2` 中的位置，计算“当秒英雄位置中心”。
  - 如果两个候选在重叠秒或边界秒的英雄位置中心距离 `<= 2500` 码，且共享英雄数 `>= 2` 或英雄 Jaccard 重合度 `>= 0.2`，则视为同一波战斗并合并。
  - 这里的英雄位置中心只使用 `npc_dota_hero_*` 英雄本体坐标，不使用召唤物、支配单位、技能命中点或 combat log 目标落点。
- 追击/续团合并：
  - 两个候选战斗间隔满足分段阈值时，如果两段战斗的英雄对英雄交互记录中存在至少 `1` 个相同英雄，则视为同一波战斗的延续并合并。
  - `0 <= time < 600`：前 10 分钟续团间隔阈值为 `<= 6s`。
  - `time >= 600`：10 分钟后续团间隔阈值维持 `<= 13s`。
  - “英雄对英雄交互”包括英雄对英雄伤害、控制和死亡信号；比较的是交互双方英雄名，不看整段战斗中心是否接近。
  - 共享交互英雄只认非远程直接交互：若某条交互中施加者与目标英雄本体距离 `> 2000` 码，则远程施加者不作为续团/追击合并的共享英雄；该条交互只保留被命中目标作为现场交互英雄。
  - 全图或超远距离技能，例如先知大招，只能作为远程贡献记录，不能单独把两个空间上独立的战斗候选合并。
  - 对共享交互英雄读取 `player_intervals2` 的逐秒英雄本体位置，从上一段结束秒到下一段开始秒逐秒计算追击中心；若逐秒中心最大跳动 `<= 3500` 码，才允许合并。
  - 合并后的事件中心用合并窗口内英雄本体逐秒位置重新计算，避免使用旧的单段技能/伤害落点中心。
  - 如果后一段原本被识别为 `GANK`，但满足上述续团条件，则把它作为上一波战斗的追击结果吸收，不再单独输出 `GANK` 标签。
- 合并后重判：
  - 任意两个战斗候选完成相邻合并、追击合并或续团合并后，必须用合并后的双方英雄数、直接参与英雄数、总伤害、死亡数和持续时间重新判定主战斗标签。
  - 若合并后满足团战条件，即使原始第一段是“小规模冲突”，最终也升级为“团战”。
  - 例如 8842468101 的 `21:36-22:14`，原始多段小规模冲突合并后双方均 5 人、直接参与人数和伤害/死亡都达到团战门槛，因此输出为“团战”。

### 12. 单杀

- 计算逻辑：
  - 单杀看击杀方：若一次死亡事件中，击杀方在死亡前 `12s` 核心交互窗口内只有 `1` 名核心击杀英雄，则输出“单杀”。
  - 不要求被击杀方只有 `1` 人；被击杀方队友如果在死亡前 `12s` 内与该击杀链有交互，会保留在 `heroes` 字段里。
  - 例如“末日使者单杀莱恩，恐怖利刃也在”：输出 `单杀`，heroes 写 `天辉：恐怖利刃、莱恩；夜魇：末日使者`。
- 输出：
  - 标记为“单杀”。
  - 当前代码还会对 `20-300s` 的早期死亡补充对线击杀上下文，根据死亡位置判断上/中/下路。
- 置信度：`0.69`

### 13. GANK

- 通用远程贡献规则：
  - 适用于 `GANK`、`小规模冲突`、`团战`。
  - 如果某条伤害、控制或 BUFF 发生时，施加者英雄本体位置 `attacker_pos` 与目标英雄本体位置 `target_pos` 的距离超过 `2000` 码，则施加者记为“远程贡献者”。
  - 距离判断只用施加者和被伤害/被 BUFF 目标在该秒附近的英雄本体坐标，不用战斗中心、技能命中点、目标落点或召唤物位置计算。
  - 如果某条信号缺少施加者或目标的英雄本体坐标，则不按远程贡献处理。
  - 远程贡献者在 `heroes` 字段中写作 `英雄（远程）`。
  - 如果同一英雄在同一事件中先有远程贡献，后续本体进场并形成直接参与，即 `首次远程贡献时间 < 首次直接参与时间`，则写作 `英雄（远程后进场）`。
  - 如果英雄先在场内直接参与，之后离开战场边缘继续输出，不写作 `远程后进场`，仍按普通参与英雄显示。
  - 远程贡献者不增加参与人数，不计入直接参与人数，也不作为 GANK/小规模冲突/团战人数判定依据。
  - 远程伤害单独写入 evidence 的 `remote_damage`，不作为主战斗伤害阈值。
- 有死亡的战斗使用死亡前 `12s` 核心交互窗口重算参与英雄：
  - 只统计死亡前 `12s` 内，与该击杀链发生英雄对英雄伤害、控制或死亡交互的英雄。
  - 更早时间被碰到、但死亡前 `12s` 没有和核心英雄互动的英雄，不计入本条死亡战斗参与人数。
  - 远程贡献仍按 `>2000` 码标远程，不参与人数计算。
  - 若死亡前 `12s` 核心交互后形成直接 `1v1`，则输出“单杀”。
  - 例如 `8829158433` 的 `01:09` 拉比克死亡，末日使者在该 `12s` 窗口内没有与寒冬飞龙/拉比克发生核心交互，因此不计入，输出为“对线击杀 / 单杀”。
- 定义：N 抓 1，即一方多名英雄在发起阶段对另一方单个英雄形成局部人数优势并进行先手/集火。
- 新计算规则：
  - GANK 按发起时刻判断，不按完整战斗窗口的最终参与人数判断。
  - 先在战斗簇内找同一目标在短窗口内被同一方至少 2 名英雄直接命中的候选。
  - 发起时刻 `t0` 取第二名不同进攻方英雄第一次命中同一目标的时刻。
  - 从 `player_intervals2` 读取所有英雄在 `t0 - 1s <= time <= t0 + 1s` 的位置。
  - 以被抓目标位置为战场中心，统计 `1600` 码内双方英雄。
  - 进攻方附近英雄数 `>= 2`，被抓方附近英雄数 `= 1`，且该英雄就是被抓目标，则判定为 `GANK`。
  - 被抓方其他英雄如果在 `t0 ± 1s` 不在战场附近，但后续 TP、走入或位移加入，只记为后续支援/反打，不取消 GANK 标签。
- 事件保留条件：
  - 被抓目标承受非远程主战斗伤害 `>= 600`，或
  - 被抓目标死亡。
- 输出：
  - 标记为“GANK”。
  - 结果字段需要写清 GANK 方、被抓目标、后续支援英雄和最终击杀结果。
  - 例如：`痛苦女王、凤凰 GANK 风暴之灵；莱恩后续 TP 支援并反打，最终天辉 1 - 1 夜魇。`
- 标签优先级：
  - GANK 优先于小规模冲突。
  - 不因为后续支援人数增加而改成小规模冲突。
  - 不因为发生在塔附近而改成塔下；塔信息可写入结果或证据。
- 置信度：`0.70`
- 详细规则文档：`GANK计算规则_v2.md`

### 14. 小规模冲突

- 当前实现：
  - 从 `combat_logs` 抽取英雄对英雄伤害、控制、死亡信号。
  - 战斗中心分为两种：
    - `center`：全窗口所有战斗信号位置的平均值，用于最终记录和部分合并判断。
    - `tail_center`：最近一段时间、最多最近 `8` 条有位置的战斗信号平均值，用于合并判断，避免边打边走时中心被早期位置拖住。
  - 相邻信号合并条件：
    - `0 <= time < 600`：时间间隔 `<= 7s`，且 `tail_center` 使用最近 `7s` 内信号。
    - `time >= 600`：时间间隔 `<= 12s`，且 `tail_center` 使用最近 `12s` 内信号。
    - 两个阶段都要求 `tail_center` 到新信号位置的游戏坐标距离 `<= 2200` 码。
  - 超过 `2200` 码时拆成独立战斗，避免跨区域消耗被统计在一起；该阈值允许同一场战斗边打边走、追击时继续累计到同一事件。
  - 双方参与英雄数都 `>= 2`。
  - 不满足团战条件时，归为“小规模冲突”。
- 事件保留条件：
  - `0 <= time < 600`：前 10 分钟按事件长度分段保留：
    - 短事件 `duration <= 12s`：保留规则不变，单个英雄承伤 `> 400` 或总伤害 `> 900`。
    - 长事件 `duration > 12s`：不再看整段总伤害，改看滑动 `8s` 爆发窗口；任意 `8s` 内总伤害 `>= 700`，或任意 `8s` 内单个英雄承伤 `>= 360`，或有英雄死亡，才保留。
    - 前 10 分钟长窗口这样处理，是为了过滤持续时间很长但强度不足的对线换血。
  - `time >= 600`：10 分钟后维持旧口径，总伤害 `>= 550` 或有英雄死亡即可保留。
- 输出：
  - 标记为“小规模冲突”。
- 置信度：`0.68`

### 15. 团战

- 当前实现：
  - 双方参与英雄数都 `>= 4`。
  - 双方直接参与英雄数都 `>= 3`。
  - 并且满足以下任一条件：
    - 总伤害 `>= 2500`
    - 英雄死亡数 `>= 2`
    - 持续时间 `>= 15s`
- 说明：
  - 参与英雄数包含被打、死亡、被控制等参与者。
  - 直接参与英雄数主要来自造成伤害或施加控制的一方英雄。
- 输出：
  - 标记为“团战”。
- 置信度：`0.74`

### 16. 肉山团

- 类型：战斗事件的上下文标签，不是独立基础事件。
- 适用范围：
  - 非 GANK
  - 非单杀
- 计算逻辑：
  - 只在 `15:00` 之后计算肉山团；`15:00` 前即使战斗靠近肉山坑，也不追加“肉山团”标签。
  - 只对 `团战` 和 `小规模冲突` 判断肉山团，不改写 `GANK` / `单杀`。
  - 读取战斗开始后的前 `3` 个秒点，即 `start, start+1, start+2`。
  - 对这 3 秒内所有参与英雄的本体坐标求平均，得到“开战前三秒参与英雄平均坐标”。
  - 该平均坐标在肉山坑 `1000` 游戏码内，才给该战斗追加“肉山团”标签。
  - 不再用任意一条伤害/控制/死亡交互点是否靠近肉山坑作为充分条件，也不用 Roshan 伤害/击杀的时间接近作为充分条件。
  - 肉山坑固定坐标：
    - 肉山上路坑：server 坐标 `(-2924.964844, 1522.657715)`，按当前 homography 换算为 raw 坐标约 `(104.0, 141.7)`。
    - 肉山下路坑：server 坐标 `(2867.646973, -3401.379395)`，按当前 homography 换算为 raw 坐标约 `(152.3, 100.5)`。
  - Roshan 被击杀后的后续追击、TP 到半图外后的战斗，即使时间接近 Roshan 击杀，也不标为肉山团。
- 输出示例：
  - `团战 / 肉山团`
  - `小规模冲突 / 肉山团`
  - 结果列补充：`肉山坑上下文：肉山上路坑战斗` 或 `肉山坑上下文：肉山下路坑战斗`

### 17. 塔下

- 类型：战斗事件的上下文标签，不是独立基础事件。
- 适用范围：
  - 非 GANK
  - 非单杀
- 计算逻辑：
  - 优先用开战前三秒的直接参战英雄位置中心判断是否在存活防御塔附近。
  - 直接参战中心会排除 `远程`、`远程后进场` 英雄，避免远处技能或后续进场把开战中心拉偏。
  - 如果开战前三秒位置缺失，则回退使用全窗口战斗中心点。
  - 当前距离阈值约为 `1000` 游戏单位。
  - 防御塔状态来自塔状态更新数据，要求该塔在战斗发生时仍存活。
- 输出示例：
  - `团战 / 塔下`
  - `小规模冲突 / 塔下`
- 结果补充：
  - 会记录是哪一方防御塔附近交战，以及塔血量等上下文。

## 四、当前体系总结

当前战术事件体系分为两层：

1. 事实类事件：直接从单类或少量数据源抽取明确事实，例如对线分路、断线、勾兵、拉野、控符、开雾、肉山击杀、肉山尝试、资源获得、带盾阵亡。
2. 战斗类事件：先聚合英雄间伤害、控制、死亡信号，再按参与人数、伤害、死亡、持续时间和场景上下文分类为单杀、GANK、小规模冲突、团战，并进一步追加肉山团、塔下等上下文标签。

后续重点是补足未落地口径，并用假设检验验证每一类口径的置信度，使置信度成为衡量口径模型好坏的标准。
